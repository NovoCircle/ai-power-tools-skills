# EA Repository SQL Schema — Gotchas Reference

Use this whenever you need to write raw SQL via `execute_sql` against an EA
repository. The schema is generated by Sparx and is **deeply inconsistent**;
guessing column names is the #1 source of "invalid column name" errors.

**Authoritative source:** `EASchema_<version>_SQLServer.sql` (shipped with EA,
checked in under `ea-mcp-server/schema/`). The MCP server parses it into
`ea_mcp_server/schema.py`. Call `describe_table("t_object")` from a tool call
to see the canonical column list for any table — cheaper than guessing.

> **Before writing SQL:** call `describe_table` for any table whose columns
> you're not 100% sure of. The MCP server now pre-validates SQL and will
> reject queries with `did_you_mean` suggestions before round-tripping to EA.

## 1. The 14 traps that bite repeatedly

1. **`t_object.Note` is SINGULAR.** Every other table uses `Notes` (plural).
   `SELECT Notes FROM t_object` fails.
2. **Tag-table column casing is UPPERCASE.** `t_attributetag`, `t_connectortag`,
   `t_operationtag` columns are `VALUE` and `NOTES`. On case-sensitive
   collations, lowercase fails. Use brackets or the exact uppercase form.
3. **There are THREE tag stores. Picking the wrong one returns empty results,
   not an error.** See §2 for the decision tree.
4. **PK naming is inconsistent.**
   * Underscore: `Object_ID`, `Connector_ID`, `Diagram_ID`, `Package_ID`,
     `TRX_ID`, `Instance_ID`.
   * No underscore: `OperationID`, `PropertyID`, `CategoryID`, `IssueID`,
     `TaskID`, `DatatypeID`, `ImageID`, `ScriptID`, `ReqID`.
   * nvarchar GUID (not int!): `XrefID`, `DocID`, `SnapshotID`, `TemplateID`,
     `PlanID`, `FileID`, `ListID`, `PhaseID`, `RuleID`.
5. **`Parent` columns differ.**
   * `t_package.Parent_ID` — underscore.
   * `t_object.ParentID` — no underscore.
   * `t_diagram.ParentID` — parent *diagram* (not package). Almost always 0.
6. **`Type` is never just `Type` on the big tables.** Use `Object_Type`,
   `Connector_Type`, `Diagram_Type`. Lookup tables (`t_objecttypes` etc.)
   do use `Object_Type` / `Connector_Type` / `Diagram_Type` as their PK
   columns too.
7. **Connector endpoints: `Start_Object_ID` / `End_Object_ID`** (underscore in
   the middle). Not `SourceID`, not `Source_ID`, not `From_ID`.
8. **`t_attribute.ID`** is the PK (not `Attribute_ID`). `Object_ID` is the FK
   back to the owning class. `t_operation.OperationID` is the PK there.
9. **`t_stereotypes` has composite PK `(AppliesTo, Stereotype)`.** The same
   stereotype name can apply to multiple metaclasses.
10. **`ea_guid` length varies.** `nvarchar(40)` on most tables but
    `nvarchar(50)` on `t_attribute`, `t_operation`, `t_operationparams`,
    `t_stereotypes`. Code that hardcodes one width silently truncates.
11. **Packages are also Objects.** Every `t_package` row has a mirror
    `t_object` row with `Object_Type = 'Package'`, and `PDATA1` holds the
    `t_package.Package_ID` *as a string*. Half of EA's own queries traverse
    via `t_object`; half via `t_package`. Write paths must keep both
    consistent — yet another reason to use COM editing tools (see §4) for
    writes, not raw SQL.
12. **Reserved-word columns must be bracketed.** `[Constraint]`, `[Default]`,
    `[Type]`, `[Order]`, `[Position]`, `[Value]`, `[Const]`, `[Path]`,
    `[Status]`, `[Key]`. `describe_table` lists the reserved columns for
    each table in `reserved_word_columns`.
13. **`t_xref` is the hidden third pillar.** Stereotype profile applications,
    MDG metadata, and many cross-references live here. PK is nvarchar(255).
    `Type`/`Client`/`Supplier`/`Description` are the discriminators.
    Whenever data "seems missing" from `t_objectproperties` or
    `t_taggedvalue`, check `t_xref` next.
14. **Binary columns blow up `SELECT *`.** Avoid `SELECT *` against
    `t_document` (BinContent), `t_snapshot` (BinContent1/2), `t_image`
    (Image), `t_stereotypes` (Metafile). Name your columns explicitly.

## 2. Tagged values — which store do I use?

**Shortcut:** call `get_element_tags(element_id=..., include_xref=True)` and
let the server query all three stores for you. The rest of this section is
for when you need to write the SQL by hand.

Three different stores, each used for different entity kinds. Picking the
wrong one returns empty (not an error), which masks the bug.

| Store | Keyed by | Used for | Notes |
|---|---|---|---|
| `t_objectproperties` | `Object_ID` (int) | **Element** tagged values (classes, requirements, actions, components, packages — anything in `t_object`) | Columns: `Property`, `Value`, `Notes`. Mixed case. Has its own `ea_guid`. This is what EA's "Tagged Values" tab on an element reads/writes. |
| `t_taggedvalue` | `ElementID` (**nvarchar(40) GUID**), `BaseClass` discriminator | Unified store EA uses internally for some entity kinds | Column is `TagValue` (not `Value`). `PropertyID` is also a GUID, not an int. `BaseClass` values: `'Class'`, `'Operation'`, `'Attribute'`, `'Connector'`, etc. |
| `t_attributetag` / `t_operationtag` / `t_connectortag` | `ElementID` (int) — = `t_attribute.ID` / `t_operation.OperationID` / `t_connector.Connector_ID` | Per-kind feature tags | Columns `VALUE`, `NOTES` (**UPPERCASE**). Don't write lowercase. |

**Rule of thumb:**
* Tags on a *Class / Requirement / Action / Package / any t_object row* → `t_objectproperties`.
* Tags on an *Attribute / Operation / Connector* → the per-kind table (`t_attributetag` / `t_operationtag` / `t_connectortag`).
* If both come up empty for an entity you know has tags → check `t_taggedvalue` with the entity's `ea_guid` as `ElementID`.

## 2.1 Schema-aware helpers — prefer these over raw SQL

The MCP server ships schema-aware tools that wrap the most error-prone
queries. Reach for these first; only drop to `execute_sql` when none fits.

| Tool | Replaces hand-written SQL against | When to use |
|---|---|---|
| `describe_table(table)` | n/a — pure introspection | Before writing any SQL against a table you're not 100% sure about. |
| `list_ea_tables()` | n/a — pure introspection | When you don't know which table holds a piece of data. |
| `get_element_tags(element_id, include_xref=True)` | `t_objectproperties` + `t_taggedvalue` + `t_xref` | Reading element tagged values without guessing which store has them. |
| `get_connectors_for_element_filtered(element_id, direction, connector_type, stereotype)` | `t_connector` with `Start_Object_ID`/`End_Object_ID` filters | Listing connectors with direction or type filters in one shot. |
| `find_stereotype_usage(stereotype, applies_to, include_xref=True)` | `t_object.Stereotype` + `t_connector.Stereotype` + `t_xref` + `t_stereotypes` | Auditing where a stereotype is used and whether it's profile-registered. |
| `list_package_tree(root_package_id, include_diagrams, include_element_counts)` | recursive `t_package.Parent_ID` walk + joins | Mapping out a sub-tree with counts in a batched query. |

## 3. The `t_xref` cookbook

`t_xref` is where MDG profile applications, stereotype assignments beyond
the free-text `Stereotype` column, and many cross-references live.

| `Type` value | What it represents |
|---|---|
| `attribute property` | Profile-defined stereotype/properties applied to an attribute |
| `operation property` | Same, for an operation |
| `element property` | Same, for an element (`Client` = element GUID) |
| `connector property` | Same, for a connector |

`Client` is the GUID of the entity the xref attaches to. `Description`
usually carries an `@TAG=…;@ENDTAG;` blob — that's where applied stereotypes
and profile-defined tagged values are stored when the entity carries an MDG
stereotype.

When "I set a tagged value with the MDG and now I can't find it in
`t_objectproperties`" — it's almost certainly in `t_xref.Description`.

## 4. Read vs. write — when to *not* use `execute_sql`

`execute_sql` goes through EA's `SQLQuery` API, which **blocks mutating SQL**.
But even if it didn't, raw-SQL writes bypass EA's in-memory model: changes
won't show up until EA reloads, related caches drift, and the
`t_object`/`t_package` mirror invariant (§1.11) can break.

For writes always use the COM editing tools:

* `start_editing_transaction` / `perform_editing_operations` / `commit_editing_transaction`
* `create_element`, `update_element`, `delete_element`, etc.
* `set_tagged_value`

`execute_sql` is for **reads, analysis, gap detection, and verification**.

## 5. Base data — what's already in a fresh repository

After `EABase_<version>_*.sql` runs (which Sparx does on every new repo),
the following tables are pre-populated *before* any user data or MDG:

* `t_objecttypes` — every UML/SysML/ArchiMate metaclass EA ships natively.
* `t_connectortypes` — every relationship type.
* `t_diagramtypes` — every diagram type.
* `t_datatypes` — datatype mappings for every supported DBMS.
* `t_stereotypes` — UML core stereotypes (custom MDGs add more, scoped by `AppliesTo`).
* `t_propertytypes`, `t_constants`, `t_cardinality`, `t_primitives`,
  `t_constrainttypes`.
* `t_efforttypes`, `t_metrictypes`, `t_risktypes`, `t_problemtypes`,
  `t_scenariotypes`, `t_testtypes`, `t_mainttypes`, `t_complexitytypes`,
  `t_statustypes`.
* `t_html`, `t_rtf` — report templates.
* `t_xrefsystem` — built-in cross-reference type catalog.
* Seed root in `t_package` and `t_object`.

If a stereotype/type query returns empty, the issue is probably *not* that
the type doesn't exist — it's that you're filtering on the wrong column or
table. `describe_table("t_objecttypes")` shows the catalog shape.
